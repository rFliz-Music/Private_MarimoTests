# Event Sytnhesis Materials Documentation


## Big Picture

This repository is an experimental environment for building interactive systems for procedural music and generative composition. The project combines Python for high-level musical algorithms with JavaScript for real-time visualization and audio playback. Marimo acts as the bridge between these two worlds, allowing Python objects and JavaScript widgets to communicate bidirectionally through AnyWidget and traitlets.

Rather than treating the notebook as a place to write isolated experiments, the repository is organized as a small software ecosystem. Python is responsible for representing musical structures (Markov chains, event sequences, timelines, probabilistic models, etc.), while JavaScript is responsible for presenting these structures interactively and rendering them in real time. The notebook itself serves primarily as the orchestration layer that instantiates objects, connects components together, and provides an interactive workspace.

---

## Three Layers

The repository can be understood as three cooperating layers.

### 1. Python (Musical Logic)

The Python side contains the compositional and mathematical logic of the system. This includes data structures such as event tuples, Markov chains, graph algorithms, generators, and any higher-level musical processes.

Python never schedules audio directly.

Instead, it responds to requests coming from JavaScript by generating musical data (timelines or event chunks), which are then transmitted through AnyWidget traitlets.

Its role is to answer the question:

> "What music should happen next?"

---

### 2. JavaScript Widgets (Realtime Runtime)

The JavaScript side is responsible for everything that must happen in real time.

This includes:

* Tone.js audio synthesis
* D3 visualizations
* user interface controls
* scheduling events
* communication with Python
* playback state

Unlike Python, JavaScript owns the concept of continuous time. It requests new musical material whenever its scheduling buffer begins to run low, allowing musical processes to continue indefinitely while Python remains purely reactive.

The JavaScript runtime is intentionally modular. Individual responsibilities such as the audio engine, stream player, timeline player, controllers, and visual components are separated into independent ES modules that communicate through clearly defined interfaces.

---

### 3. Marimo (Glue Layer)

Marimo connects the Python and JavaScript worlds.

Each notebook creates Python objects, instantiates widgets, registers callback functions, and exposes traitlets that allow both sides to exchange information.

Ideally, notebooks remain lightweight.

Their purpose is not to implement algorithms, but to assemble systems from reusable Python and JavaScript components.

---

## Widget Architecture

Every widget is a small application.

Each widget consists of:

* Python widget class
* JavaScript entry point
* optional CSS
* reusable JavaScript modules
* optional Python callbacks

The JavaScript entry point is intentionally small. Its job is to assemble reusable components rather than implement large amounts of logic directly.

For example, a sampler widget may construct:

* AudioEngine
* EventStreamPlayer
* EventTimelinePlayer
* SamplerController

rather than containing all of this logic in a single file.

---

## Build System

The repository uses Node.js and esbuild to bundle JavaScript modules into single widget files that Marimo can load.

Source files live under the widget source directory and may freely import:

* project modules
* Tone.js
* D3
* other npm packages

Running the build process resolves every dependency and produces a self-contained JavaScript bundle (and optional CSS) inside the distribution directory.

Python never loads the source modules directly; it only reads the generated bundle.

The build artifacts are therefore considered generated files rather than source code.

---

## Philosophy

The long-term goal is to separate concerns as clearly as possible.

Python should express algorithmic musical ideas.

JavaScript should express interaction and real-time behavior.

Marimo should express pedagogical exposition.

Whenever a component grows beyond a single responsibility, it should become its own module rather than increasing the complexity of existing files.

The repository should evolve as a collection of small, composable musical building blocks that can be assembled into increasingly sophisticated instruments, visualizations, and compositional systems.

#  Brief API Documentation

## Audio Engine: Event Scheduling

The audio engine is responsible for translating symbolic musical events generated by Python into real-time audio using Tone.js. It currently supports two distinct playback models:

* **Timeline playback**, for finite precomputed sequences.
* **Event stream playback**, for potentially infinite procedural music.

Although both ultimately schedule events on `Tone.Transport`, they differ fundamentally in how musical material is obtained.

---

## EventTimelinePlayer

`EventTimelinePlayer` plays a complete musical timeline that already exists in memory.

A timeline consists of events of the form:

```text
[pitch, velocity, duration, start_time]
```

where `start_time` is an absolute time measured from the beginning of the transport.

Playback proceeds as follows:

1. Stop and clear the current transport.
2. Schedule every event at its specified start time.
3. Start the transport.

Since the complete musical sequence is already known, no communication with Python occurs during playback.

This scheduler is appropriate for finite compositions, exported MIDI sequences, or any musical process whose entire timeline can be computed beforehand.

---

## EventStreamPlayer

`EventStreamPlayer` is designed for procedural music whose future cannot be computed all at once.

Instead of scheduling an entire composition, it repeatedly requests small chunks of musical events from Python while playback is already in progress.

This allows musical processes such as Markov chains, probabilistic generators, or interactive systems to run indefinitely without ever constructing a complete timeline.

---

## Event Representation

Streaming events are represented as

```text
[pitch, velocity, duration, delta]
```

Unlike timeline playback, events do not contain absolute start times.

Instead:

* **duration** specifies how long the note should sound.
* **delta** specifies how much logical musical time should advance before the next event.

Separating these two quantities makes it possible to express:

* legato
* staccato
* overlapping notes
* sustained notes
* simultaneous voices

without changing the scheduling algorithm itself.

---

## Scheduling Strategy

The scheduler maintains a single variable:

```text
scheduledUntil
```

which represents the end of all currently scheduled musical material.

Whenever a new chunk arrives from Python:

1. old Transport events are cleaned up
2. every incoming event is scheduled beginning at `scheduledUntil`
3. each event advances local time by its `delta`
4. `scheduledUntil` is updated to the end of the newly scheduled region

The scheduler therefore never needs to know the complete future.

It only needs to know where the currently scheduled future ends.

---

## Chunk-Based Streaming

Rather than requesting events one at a time, the scheduler requests fixed-size chunks.

```text
chunkSize
```

controls how many events Python should generate for each request.

Chunking significantly reduces communication overhead between JavaScript and Python while allowing the musical process to remain effectively infinite.

---

## Lookahead Buffer

To avoid audio dropouts, playback always keeps a small amount of future material scheduled.

This is controlled by

```text
lookahead
```

measured in seconds.

Whenever the amount of scheduled future falls below the lookahead horizon, JavaScript requests another chunk from Python.

The scheduler therefore behaves similarly to an audio streaming buffer.

```
current time                scheduledUntil
     |-----------------------------|
            lookahead
```

As playback advances, the lookahead window moves forward continuously.

---

## Refill Timer

Instead of constantly polling the transport, the scheduler computes exactly when another request will become necessary.

After scheduling a chunk, it calculates

```
scheduledUntil
− lookahead
− current transport time
```

and installs a single timer that will wake the scheduler at precisely that moment.

When the timer fires, JavaScript requests another chunk from Python.

Only one refill timer exists at any time.

This makes the scheduling system event-driven rather than continuously polling, minimizing unnecessary work while guaranteeing that future musical material is requested before the playback buffer becomes empty.

---

## Event Cleanup

Tone.Transport retains scheduled callbacks even after they have executed.

Without explicit cleanup, an infinitely running musical process would accumulate Transport events indefinitely.

To prevent this, every scheduled callback ID is stored.

Whenever a new chunk is scheduled, callbacks whose scheduled execution time lies safely in the past are removed from the Transport.

This keeps memory usage approximately constant regardless of how long the system runs.

---

## Python Interaction

The JavaScript scheduler owns time.

Python owns musical decisions.

Whenever additional musical material is required, JavaScript emits

```
request_chunk
```

through the widget trait system.

Python responds by generating a new list of symbolic musical events.

JavaScript immediately schedules those events into the existing future buffer.

This separation keeps Python entirely stateless with respect to real-time scheduling while allowing arbitrary compositional algorithms to generate future musical material on demand.
